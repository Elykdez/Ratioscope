# Local LLM chat runtime (Unity Sentis)

This project runs a local LLM checkpoint as a text-only chat model entirely inside
Unity, on Unity Inference Engine (Sentis, `com.unity.ai.inference` 2.6.1). There is no
Python service at runtime; Python is used only as offline export tooling. All inference
stays local and offline.

Two production checkpoints ship as four model/backend options (`LlmSystemOption`). The CPU and
GPU variants share the same fixed-context artifact; only the Sentis backend changes. A
`Tiny = -1` profile is reserved for tests and editor iteration; its 512-token random-weight
vocabulary cannot consume a real ChatML prompt, so `ChatService` refuses such prompts (see
"Vocabulary guard") instead of letting Sentis read out of bounds:

| Option | Checkpoint | Window | Backend | Artifact |
| --- | --- | --- | --- | --- |
| `Cpu1_7B_2048` | `Qwen/Qwen3-1.7B` | 2048 | CPU | `Llm_Decode_2048.sentis` |
| `Gpu1_7B_2048` | `Qwen/Qwen3-1.7B` | 2048 | GPUCompute | `Llm_Decode_2048.sentis` |
| `Cpu4B_4096` | `Qwen/Qwen3-4B-Instruct-2507` | 4096 | CPU | `Llm_Decode_4096.sentis` |
| `Gpu4B_4096` | `Qwen/Qwen3-4B-Instruct-2507` | 4096 | GPUCompute | `Llm_Decode_4096.sentis` |

`Assets/Bundles/Settings/LlmSystemSettings.asset` controls automatic selection. Unity
reports adapter capacity rather than free VRAM, so the default configuration reserves
0.5 GiB first. It selects GPU 4B on 16 GB-class cards or larger (15.5 GiB usable),
GPU 1.7B on 8 GB-class cards (7.5 GiB usable), and CPU 1.7B below that or when no GPU
is available; the half-GiB thresholds account for the reserve. CPU 4B is manual.
(The 4B threshold was originally 7.8 GiB, which handed any 8 GiB card a graph that
cannot fit: a full `Gpu4B_4096` session measures ~17.5 GB GPU memory, so expect some
shared-memory spill on exactly-16 GB cards.)

Measured on an RTX 3070 8 GiB: the uint8 4B artifact saturates the card even at small
contexts (weights plus per-schedule dequantize transients peak ~7.8 GiB), so 4B-on-GPU
requires a bigger card; on CPU it runs ~4.5 s/token. The 1.7B/2048 default fits with
real margin. Both checkpoints are dense Qwen3 transformers (Apache 2.0) sharing a
byte-identical `tokenizer.json`. The pipeline and all names are model-agnostic (`Llm*`):
swapping checkpoints means re-running the exports below against a different
`--model-dir`, not renaming code. Any standard dense causal LM with a GPT-2-style
byte-level BPE tokenizer (Qwen, Llama 3, Phi, SmolLM) fits the pipeline as-is;
hybrid architectures (Qwen3.5+ Gated DeltaNet, SSM/Mamba) do not survive ONNX->Sentis.
Qwen3-1.7B is a thinking-capable checkpoint, so its presets set
`ChatServiceOptions.SupportsThinking`. The Cortex UI exposes a **Thinking** toggle under
**Settings** and leaves it off by default so reasoning cannot consume the fixed context before
a visible answer begins. API callers opt in with `ChatGenerationOptions.EnableThinking = true`;
the `<think>` stream is then exposed separately from visible answer text. With thinking disabled,
the prompt pre-closes the think block like `apply_chat_template(enable_thinking=False)`.
Non-thinking checkpoints ignore this flag.

The Unity API is `ChatService` (`Assets/Scripts/Modules/AI`).

## Architecture

Checkpoints live in `Tools/ImportedModel` by default; a `MODEL_ROOT` environment variable
relocates them to any other directory.

```text
Tools/ImportedModel/Qwen3-1.7B                (HF checkpoint, 3.4 GB bf16)
Tools/ImportedModel/Qwen3-4B-Instruct-2507    (HF checkpoint, 7.5 GB bf16)
        |
        |  Tools/LlmChat/export_llm_sentis.py --decode-only    (offline, Python)
        v
Tools/ExportedModel/Llm_Decode_2048.onnx     (1.7B fp32, fixed 2048 context;
Tools/ExportedModel/Llm_Decode_4096.onnx      4B fp32, fixed 4096 context;
        |                                     each with .weights/*.bin external data,
        |                                     linear weights pre-transposed)
        |  Unity menu: Tools/Hypocycloid/Sentis/ONNX to Sentis
        v
Assets/StreamingAssets/Sentis/Llm_Decode_2048.sentis (uint8 KV decode, CPU/GPU 1.7B)
Assets/StreamingAssets/Sentis/Llm_Decode_4096.sentis (uint8 KV decode, CPU/GPU 4B)

Tools/ImportedModel/Qwen3-1.7B/tokenizer.json (byte-identical across both checkpoints)
        |
        |  Tools/LlmChat/export_llm_tokenizer.py         (offline, Python)
        v
Assets/StreamingAssets/Llm/llm_tokenizer.bin             (3.4 MB vocab+merges binary)
Assets/Editor/Tests/Fixtures/llm_tokenizer_fixtures.json (HF reference fixtures)
```

Runtime pieces (all C#, no external process):

- `LlmTokenizer` - GPT-2-style byte-level BPE. Loads the binary above (which carries
  the checkpoint's pre-tokenization split regex) and reproduces the Hugging Face
  fast-tokenizer pipeline exactly (verified against fixtures generated by
  `AutoTokenizer`, including CJK, emoji, digits, and contractions).
- `ChatTemplate` - builds the ChatML prompt string
  (`<|im_start|>system\n...<|im_end|>\n<|im_start|>user\n...<|im_end|>\n<|im_start|>assistant\n`),
  mirroring the checkpoint's `chat_template` for text-only system/user/assistant chats.
  Tool calling is not implemented.
- `ChatService` - loads the `.sentis` model, owns one Sentis `Worker`, and runs
  the autoregressive loop: greedy decoding at `Temperature <= 0`, otherwise
  temperature + top-k + top-p sampling. Generation stops at `<|im_end|>` (151645),
  `<|endoftext|>` (151643), `MaxNewTokens`, or a full context window.
- `ChatStream` - submits each GPUCompute forward with `Worker.Schedule()` so the graph
  is one command-buffer batch. CPU keeps `ScheduleIterable()` for frame budgeting.
  GPU graph telemetry emits one status marker per batch; `ForwardEvaluated` fires only
  after logits are readable. Neither path splits a GPU forward for visualization.

Tokenizer loading is owned by `ChatService`. The serialized `LlmSystemSettings` asset
resolves its configurable tokenizer-relative path under `Application.streamingAssetsPath`
when it creates `ChatServiceOptions`. Both `ChatService.Start()` and `StartAsync()` call
`LoadArtifacts()`, which verifies the file exists and constructs `LlmTokenizer` before
loading the Sentis model. Because the file is under `StreamingAssets`, Unity copies it
into the player build and `Application.streamingAssetsPath` resolves its deployed path.

## Fixed context and KV cache

The original fallback graph (`Llm_Text_*`) has no KV cache: inputs `input_ids` and
`attention_mask` at the full window, last-position logits `[1, 1, 151936]` out. It
remains supported as an optional export, but a full-window forward per token is
impractical at these window sizes.

The cached export keeps Sentis-compatible static shapes while avoiding a full-window
forward for every generated token:

- `Llm_Decode_<ctx>` accepts one token, a `[1,ctx]` attention mask, one position id, and
  one fixed `[1,kv_heads,ctx-1,128]` K/V pair per transformer block, returning logits plus
  updated same-shape pairs. The 1.7B graph has 28 blocks, the 4B graph 36; both use 8 KV heads.
- `Llm_Prefill_<ctx>` (optional export) accepts `[1,ctx-1]` and returns logits plus one
  K/V pair per layer, for validation and future cold-start optimization.
- `ChatService` uses the decode graph as the single persistent worker. Prompt tokens
  are ingested frame-by-frame, then only the new token is evaluated during generation,
  so only one weight set is ever resident.
- Cache tensors remain on their execution backend between schedules. Full-cache size:
  ~448 MiB (1.7B at 2048) and ~1.13 GiB (4B at 4096). `Worker.CopyOutput` copies each
  `present_*` output into the service-owned `past_*` tensor on the same backend;
  GPUCompute no longer downloads and re-uploads the cache. Only the ~0.58 MiB logits
  output is read back for CPU sampling. Worker-owned outputs are never retained across
  schedules.
- VRAM reality on an 8 GiB card (RTX 3070, measured): the 4B artifact peaks ~7.8 GiB
  even at 128 context because per-schedule dequantize transients ride on top of the
  4.15 GiB resident weights; adding cache growth crashes the editor by spilling GPU
  heaps into system commit. Hence 4B runs on CPU on this class of hardware and
  `Gpu4B_4096` requires a bigger card. fp16 cache I/O would halve cache cost but
  Sentis 2.6.1 has no fp16 graph inputs/outputs.
- Message text remains authoritative. If a new prompt extends the cached token prefix,
  only its suffix is ingested. If history was trimmed or edited, the cache is cleared
  and rebuilt from the remaining messages.

The prompt+reply budget equals the option's window (2048 by default). KV caching
improves per-token work and multi-round reuse; it does not by itself increase the
exported context length.

`ChatService.InspectContext` exposes the exact retained messages, formatted prompt,
prompt ids, occupancy, available reply tokens, and any complete turns that physically
cannot fit. Fitting history is no longer discarded merely to reserve the requested
`MaxNewTokens`; generation instead uses the remaining window space.

`CortexChatController` never compacts automatically. A normal reply uses the remaining
window space and the runtime may hard-trim complete old turns only when the prompt itself
cannot fit. The authored `COMPACT` button explicitly runs an asynchronous greedy pass on
the active model to compress the oldest complete turn plus the existing memory summary.
The generated memory is stored as a system message and the source turn is removed.
Rewriting history intentionally clears and rebuilds the KV cache.

## Vocabulary guard

Sentis's Burst `Gather` resolves the embedding row with raw pointer arithmetic and only folds
negative indices; it never checks the upper bound. A token id past the exported vocabulary
therefore reads unmapped memory and takes the process down with no managed exception - in a
player build, an instant crash to desktop. The shipped tokenizer emits ids up to 151,935, so
pairing it with the 512-vocab `Tiny` artifact makes every real prompt fatal.

Each profile declares the vocabulary its artifact was exported with (`vocabularySize` in
`LlmSystemSettings.asset`; 151936 for both production checkpoints, 512 for `Tiny`) and
`ChatService.ValidatePrompt` rejects any prompt id outside it. Sampled ids come from the
model's own logits and are in range by construction, so the prompt is the only way an
out-of-range id can reach the graph. Selecting a mismatched artifact now fails with
`START FAILED` in the chat status bar rather than killing the player.

`ChatServiceOptions.VocabularySize` defaults to 0, which disables the check; every profile
path sets it, and `LlmModelConfiguration.CreateOptions` throws when it is unconfigured.

## Weight quantization (uint8)

The ONNX is exported as float32 and quantized to uint8 inside the Unity converter
(`OnnxProcessor.QuantizeWeightsInPlace`), one constant at a time: each weight
consumed by `MatMul`/`Gather`/`Dense`/`Transpose` gets a per-tensor scale/zero-point,
a packed 1-byte-per-weight constant, and a `DequantizeUint8` layer in front of its
consumer. Two Sentis quirks the converter works around:

- Sentis's stock `ModelQuantizer` leaves a 4-bytes-per-weight backing array on byte
  constants, which would serialize uint8 at float32 size; `PackByteConstant` repacks
  to exactly 1 byte per weight.
- The dynamo ONNX export leaves every linear weight behind a `Transpose` node, whose
  op was not in the quantizable-consumer set - without the `Transpose` entry (and the
  exporter's transpose-folding pass) most weights silently stay fp32. This was also
  the source of the unexplained "fp32 residue" in the predecessor artifact.

The exporter additionally folds `Constant -> Transpose([1,0]) -> MatMul` weights into
pre-transposed initializers (`fold_transposed_weight_initializers`), removing ~250
runtime layers, and splits any initializer above Sentis's 500 MB constant limit into
offset-zero chunk files (the tied embedding/lm_head is rebuilt per chunk as
`Gather`/`MatMul` + `Concat`).

Because `tie_word_embeddings` is true, the embedding matrix feeds both a `Gather` (token
lookup) and a `Transpose -> MatMul` (lm_head), and `split_large_matrix_initializer`
materializes both forms: 512-column `Gather` chunks plus 16384-row pre-transposed
projection chunks. The tied matrix is therefore stored twice. That is why the 1.72B
checkpoint yields ~1.9B stored values (7.57 GB of fp32 external data before conversion,
~1.9 GB as uint8) rather than ~1.7 GB. Each chunk is a separate constant and gets its own
scale, so the largest tensor picks up crude group-wise quantization for free.

Resident vs transient, 1.7B at 2048: ~1.9 GB of uint8 weights stay resident, while each
`DequantizeUint8` allocates a full-size fp32 copy of its weight for the schedule - the
largest being one `151936 x 512` embedding chunk at ~311 MB. Those transients, not the
weights, are what makes headroom tight on small cards; see the VRAM bullet under
"Fixed context and KV cache" for measured numbers. Half-precision execution would halve
both the transients and the ~448 MiB cache, on the order of 400 MB against a ~2.5-3 GB
footprint - real, but not the difference between fitting and not fitting on 8 GiB.

Per-tensor uint8 is coarser than llama.cpp-style group quantization; validate chat
quality after re-exporting a new checkpoint.

Quantization shrinks the artifact and the resident weight bytes only; every arithmetic
kernel still computes in fp32. Inference Engine 2.6.1 tensor storage is limited to
float32/int32/int16/uint8 (`NativeTensorArray.DataType`) with no half-precision
execution path. The converter's `Float16` mode buys nothing as implemented: Sentis
pads `Short` data into int32 buffers, its `QuantizeConstantsPass` leaves a
4-bytes-per-weight backing array, and `PackByteConstant` repacks only uint8 - so a
Float16 artifact serializes and resides at full fp32 size behind a per-schedule
`Cast`, with only per-weight accuracy improving over per-tensor uint8. Exporting fp16
ONNX from the Python tools ends the same way: the importer casts fp16 initializers to
fp32 because the kernels are fp32-only. Weights that stay 16-bit or 4-bit through
execution (GGUF-style) require a different runtime such as llama.cpp; within Sentis
the only VRAM levers are checkpoint size and window.

## Regenerating everything from scratch

The downloaded checkpoints, exported ONNX, and `.sentis` artifacts are regenerable build
products and are gitignored (`Tools/ImportedModel/`, `Tools/ExportedModel/`, `*.sentis`).
The tokenizer binary and fixtures are committed.

`Tools\LlmChat\build_llm_model.bat` covers steps 1 and 4 for the production graphs -
it creates the venv when missing, refuses to run blind while the editor is open, and
skips graphs that already exist. The steps below are the full manual rebuild, including
the tokenizer and tiny-model artifacts the batch script does not touch. Checkpoints are
read from `Tools\ImportedModel` unless a `MODEL_ROOT` environment variable overrides it.

```powershell
# 1. One-time Python env (requires system Python 3.12)
python -m venv --system-site-packages Tools\LlmChat\.venv
Tools\LlmChat\.venv\Scripts\python.exe -m pip install -r Tools\LlmChat\requirements.txt

# 2. Tokenizer binary + Hugging Face reference fixtures (tokenizer.json is
#    byte-identical across both checkpoints; either --model-dir works)
Tools\LlmChat\.venv\Scripts\python.exe Tools\LlmChat\export_llm_tokenizer.py `
  --model-dir Tools\ImportedModel\Qwen3-4B-Instruct-2507 `
  --output Assets\StreamingAssets\Llm\llm_tokenizer.bin `
  --fixtures Assets\Editor\Tests\Fixtures\llm_tokenizer_fixtures.json

# 3. Tiny random-weight validation model (seconds; tests intentionally stay at
#    128 context so the fast suite exercises the same code with a small window)
Tools\LlmChat\.venv\Scripts\python.exe Tools\LlmChat\export_llm_sentis.py `
  --model-dir Tools\ImportedModel\Qwen3-4B-Instruct-2507 `
  --output Tools\ExportedModel\Llm_Tiny_Text_128.onnx --tiny --verify-onnxruntime

# 4. The two production decode graphs (each needs free commit roughly twice its
#    fp32 weight size; close a heavy editor first). The 4B export takes ~10 minutes.
Tools\LlmChat\.venv\Scripts\python.exe Tools\LlmChat\export_llm_sentis.py `
  --model-dir Tools\ImportedModel\Qwen3-1.7B `
  --output Tools\ExportedModel\Llm_Decode_2048.onnx `
  --decode-only --context-length 2048
Tools\LlmChat\.venv\Scripts\python.exe Tools\LlmChat\export_llm_sentis.py `
  --model-dir Tools\ImportedModel\Qwen3-4B-Instruct-2507 `
  --output Tools\ExportedModel\Llm_Decode_4096.onnx `
  --decode-only --context-length 4096

# 5. Optional uncached fallback graph (impractical to run at these window sizes;
#    export only if you need the no-cache path for debugging)
Tools\LlmChat\.venv\Scripts\python.exe Tools\LlmChat\export_llm_sentis.py `
  --model-dir Tools\ImportedModel\Qwen3-1.7B `
  --output Tools\ExportedModel\Llm_Text_2048.onnx --context-length 2048
```

Then in Unity:

- Open `Tools/Hypocycloid/Sentis/ONNX to Sentis`, select any exported ONNX, and click
  `Convert`. The output is written to `Assets/StreamingAssets/Sentis` with the same base name.
- The Qwen-friendly defaults select `Llm_Decode_2048.onnx`, packed uint8 weights, and
  skip the memory-heavy Sentis graph optimizer. Disable `Skip graph optimization` for
  smaller models when normal Sentis optimization and validation are preferred.
- Repeat the same generic conversion for each system-option graph and any optional
  text, prefill, or tiny ONNX files. The default option prefers
  `Llm_Decode_2048.sentis` when it exists and otherwise falls back to
  `Llm_Text_2048.sentis`; the 4B options load their decode artifact directly.

`SentisStreamingModelWriter` serializes the converted model in 256 MB weight segments so
neither conversion nor loading ever materializes a multi-gigabyte transient buffer.

Checkpoint loading note: the bf16 -> fp32 conversion inside
`from_pretrained(dtype=float32)` crashes natively (0xC0000005) in this environment;
the exporter loads bf16 and casts with `model.float()` instead. The export also dies
with the same code when Windows free commit is low - the Unity editor can pin tens of
GB of leaked native memory after running a large model; restart it first.

## Unity API

```csharp
// systemSettings is the serialized Assets/Bundles/Settings/LlmSystemSettings asset.
// Its profiles own artifact names, backends, thinking support, and uncached block counts.
ChatServiceOptions options = systemSettings.CreateServiceOptions(
    LlmSystemOption.Gpu1_7B_2048
);

using ChatService chat = new(options);
chat.Start(); // optional; Chat() starts lazily

ChatResult result = chat.Chat(
    new[]
    {
        ChatMessage.System("You are concise."),
        ChatMessage.User("Reply with exactly PONG."),
    },
    new ChatGenerationOptions
    {
        MaxNewTokens = 8,
        Temperature = 0f,
        EnableThinking = false, // Cortex normal chat takes this from its Settings toggle
    }
);

Debug.Log(result.Content);        // parsed reply, control tokens removed
Debug.Log(result.Thinking);       // <think>...</think> block, if the checkpoint emits one
Debug.Log(result.GeneratedTokens);
Debug.Log(result.FinishReason);   // StopToken, TokenLimit, ContextLimit, or Cancelled
```

`Chat` is synchronous and blocks the calling thread for the whole generation; use it
from worker threads, editor tooling, or tests. `GenerateTokens(promptIds, generation,
stopTokens)` exposes the raw id-level loop (pass an empty `stopTokens` collection to
disable early stopping). Keep one service alive per model; `Start()` reloads nothing
when already ready. Dispose the service to release the Sentis worker.

`ChatResult.IsComplete` is true only when the model emitted a configured stop token. Reaching
`MaxNewTokens` or filling the fixed context is reported separately instead of looking like a
normal completion. The Cortex scene sets its reply limit to `0`, meaning all context space left
after the prompt; the runtime still has the model's 2048/4096-token window as a hard safety cap.

## Streaming and Cortex view

`BeginChat` keeps the blocking `Chat` API unchanged and returns a `ChatStream`. Drive
the stream once per frame. CPU `Tick` advances only as many Sentis graph operations as fit
inside the supplied time budget; GPU `Tick` submits the whole forward and polls its output
on later frames. Only one stream may use a `ChatService` worker at a time.

```csharp
ChatStream stream = chat.BeginChat(messages, generation);
stream.ForwardEvaluated += cortex.OnForward;
stream.TokenSampled += metrics =>
{
    Debug.Log($"{metrics.Text}: H={metrics.Entropy:0.00}, {metrics.ElapsedSeconds:0.0}s");
    cortex.OnToken(metrics);
};
stream.Completed += result => Debug.Log(result.Content);

void Update()
{
    stream?.Tick(8); // milliseconds of inference work this frame
}
```

With a cached decode model, prompt suffix ingestion also raises `ForwardEvaluated` but
does not raise fake token events; `TokenSampled` still means the model committed to an
assistant token. `TokenMetrics` contains the sampled id and decoded text, full-softmax entropy, top-k
ids/text/probabilities, forward-pass duration, and prompt/window occupancy. Greedy and
sampled decoding share the same top-k selection. `BeginChat` trims the oldest complete
non-system turns only when the prompt itself cannot fit the fixed window; it throws when
the system prompt and newest turn cannot fit.

`Assets/Scenes/Stage.unity` is the product scene. Its existing Stage camera, canvas,
EventSystem, updater, and other scene components remain authoritative; an instance of
`Assets/Bundles/Prefabs/UI/Components/CortexChat.prefab` is permanently stretched over
`MainCanvas/Mainframe`. The prefab owns the complete chat UI hierarchy and all serialized
component references. Runtime code only creates message-bubble instances from
`CortexMessageBubble.prefab` and the inference heat texture/material.

The status bar's circular HDR context button shows live prompt/window occupancy. Clicking
it opens the authored context panel, which displays the exact retained roles and text,
available reply space, pending hard trims, and the generated working-memory message. Its
`COMPACT` button requests one manual compaction pass. It closes the context panel and
disables sending until the pass completes; replies never trigger compaction automatically.

The `LlmSystemSettings` asset owns the matrix width, token-strip height, pulse heat, decay,
candidate base heat, entropy smoothing, and palette scale; its model profiles own the uncached block
count fallback. Total matrix height is `blockCount * 2 + tokenRows`. `ChatService` derives
the block count from paired K/V inputs (28 for 1.7B, 36 for 4B) and uses the configured
fallback only for uncached graphs. Each block owns an attention row and an MLP row. A completed
forward pulses all rows because these dense Qwen checkpoints execute every block; this is
structural participation, not activation magnitude or per-block timing. Top candidates
flare on the deterministic token surface, entropy shifts the palette, and heat decays.
Hovering shows block/branch/stage, heat, token data, latest candidates, and entropy. During model loading,
`CortexChatController` publishes loading lifecycle events. The scene-authored
`CortexHudLoadingBinder` routes those events to the outer `Mainframe/Hud/CortexLoading`
instance of `Overlay.prefab`, keeping HUD presentation out of the chat controller.
`CortexLoadingRing.shader` renders the animated loading ring. The `LlmSystemSettings`
asset serialized on the controller prefab chooses the hardware tier by default;
disable `Auto Select System Option` to use the serialized manual option. Select `Tiny`
there only for editor/test iteration; automatic hardware selection never chooses it. When the
selected checkpoint supports thinking and the config toggle is enabled, the controller streams
its reasoning into a muted `thinking` bubble and its user-visible response into a separate `ai`
bubble. Context compaction disables thinking.

## Tests

`Assets/Editor/Tests/LlmChat.test.cs` (EditMode, `Hypocycloid.Editor.LlmChatTests`):

- Tokenizer encode/decode and special-token ids are verified against fixtures produced
  by the Hugging Face tokenizer itself, covering ASCII, multi-space runs, newlines,
  Chinese, emoji, digit splitting, and contractions.
- The chat template is verified against `apply_chat_template` output, both as text and
  as token ids, including the prompt form that disables thinking for utility generations.
- The streaming response parser is checked while the `<think>` tag is partial, while
  reasoning is in progress, after the final answer begins, and for direct answers.
- `TinyModelGreedyGenerationIsDeterministic` runs the full Sentis decode loop on the
  tiny model (real Qwen3 graph shape, uint8-quantized like the full artifact, random
  weights, 512-token vocab) and checks determinism and bounds.
- Metrics math is checked against exact synthetic-softmax values; the heat grid is
  checked for complete 28/36-block coverage, deterministic mapping, and monotonic decay.
- `TinyModelStreamMatchesBlockingGreedyPathAndEmitsTelemetry` verifies layer/token
  events and exact greedy parity with the blocking path.
- `TinyKvCacheRetainsPrefixAcrossTurnsAndMatchesRebuild` verifies that retained-prefix
  continuation exactly matches rebuilding the cache from the full extended prompt.
- `TinyKvCacheStreamMatchesBlockingGreedyPath` verifies cached streaming/blocking parity.
- `TinyKvCacheGpuStreamUsesBatchedSchedulingAndMatchesBlockingPath` verifies GPU
  streaming/blocking parity, same-device cache updates, and one forward event per batch.
- `FullModelAnswersPong` is `[Explicit]`: it loads the production decode artifact
  (1.7B at 2048) on the default backend, runs the streaming path, reports token rate,
  and asserts the deterministic reply contains `PONG`.

The GPU cache regression and two CPU cache tests pass against the converted tiny decode
fixture. After removing cache staging and per-layer GPU command-buffer flushes, the
production `FullModelAnswersPong` stream passed on an RTX 3070 across measured runs at
3.24-4.03 token/s (0.248-0.308 s/token mean), compared with the reported pre-fix rate of
about 0.5 token/s. Post-structure-map validation repeated at 3.24 and 3.25 token/s; this
benchmark does not instantiate the matrix view.

The imported 1.7B/2048 decode artifact is 2,032,789,960 bytes (1.89 GiB); the
4B/4096 artifact is 4,412,880,256 bytes (4.11 GiB). A clean 4B load check reported
75 inputs, 73 outputs, and 5,176 layers. On the RTX 3070, automatic selection chose
`Gpu1_7B_2048`, reached ready in 15 seconds, and the live controller answered the
8-token greedy smoke prompt with `PONG.`. Running the whole `LlmChatTests` group from
the Test Runner or MCP may select the explicit full-model test once the model exists;
select the non-explicit tests individually for a fast-only pass.

## Known limits

- The window (2048 by default) is shared by prompt and reply. Manual model-generated
  compaction preserves durable facts but is lossy; a system prompt plus newest turn
  that cannot fit still throws.
- Each option has a fixed exported context and rebuilds its cache after compaction or
  hard history trimming. A rebuild re-ingests the retained history one token per
  schedule, so compacting a long history is slow.
- GPUCompute retains and copies the full K/V cache on-device, but the static decode
  graph still copies ~448 MiB for the default option after every forward.
- The 4B options need either the CPU backend (~4.5 s/token measured) or a large GPU:
  the uint8 4B artifact peaks ~7.8 GiB VRAM even at tiny contexts, and a full 4096
  session measures ~17.5 GB. Automatic selection requires a 16 GB-class card
  (15.5 GiB usable), which still spills a little into shared memory at full window.
- The 1.7B checkpoint answers with lower quality than the 4B; it is the tradeoff for
  a 2048 window plus GPU speed on 8 GiB hardware.
- Per-tensor uint8 quantization trades some answer quality for a 4x smaller artifact.
- Tool calling, images, and audio are not implemented. Thinking is shown only when its config
  toggle is enabled and the selected checkpoint declares support;
  Qwen3-4B-Instruct-2507 is non-thinking.
- Consecutive same-role messages are emitted as-is, not merged.
